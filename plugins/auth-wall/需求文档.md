# Auth Wall — 需求文档

> 版本：v0.1.0
> 仓库：[yangfei957/vibe-kanban-plus](https://github.com/yangfei957/vibe-kanban-plus)

---

## 1. 项目概述

### 1.1 背景

[Vibe Kanban](https://github.com/yangfei957/vibe-kanban) 是一款本地看板工具，默认以无认证模式运行，适用于个人单机使用。然而当用户将其部署到服务器或公网环境时，缺乏访问控制将带来安全风险。

Auth Wall 旨在提供一个**轻量级、即插即用的密码认证网关**，为 Vibe Kanban（或任何基于 Axum 的 Web 应用）增加一层密码保护。

### 1.2 项目目标

- 以最小侵入方式为已有 Web 应用增加密码保护
- 独立于 Vibe Kanban 主仓库，可独立开发、测试和发布
- 安全性优先：密码学最佳实践，防暴力破解
- 零配置即可运行：未设置密码时自动放行

### 1.3 目标用户

| 用户角色 | 使用场景 |
|----------|----------|
| 个人开发者 | 将 Vibe Kanban 部署到个人 VPS 或 NAS，防止未授权访问 |
| 小型团队 | 在内网环境共享 Vibe Kanban，通过共用密码控制访问 |
| Axum 开发者 | 为自己的 Axum Web 应用快速添加密码保护层 |

## 2. 功能需求

### 2.1 需求列表

| 编号 | 需求名称 | 描述 | 优先级 |
|------|----------|------|--------|
| FR-01 | 全局请求拦截 | 对所有页面和 API 请求进行认证检查 | P0 |
| FR-02 | 独立登录页面 | 提供内嵌的 HTML 登录页面，不依赖任何前端框架 | P0 |
| FR-03 | 安全密码存储 | 使用 Argon2id 算法哈希存储密码，禁止明文 | P0 |
| FR-04 | 暴力破解防护 | 基于客户端 IP 的失败尝试计数与锁定机制 | P0 |
| FR-05 | Cookie 会话管理 | 通过 HttpOnly Cookie 维护登录状态 | P0 |
| FR-06 | Feature 门控集成 | 以 Cargo feature 方式集成，不启用时零代码影响 | P0 |
| FR-07 | 优雅降级 | 未设置密码时自动放行所有请求 | P1 |
| FR-08 | 环境变量配置 | 所有可调参数均可通过环境变量覆盖 | P1 |
| FR-09 | Secure Cookie | 支持 HTTPS 环境下设置 Cookie 的 Secure 标记 | P2 |
| FR-10 | 完全独立 | 不依赖 Vibe Kanban 内部任何 crate，可在其他 Axum 项目复用 | P1 |
| FR-11 | CLI 密码管理 | 提供交互式命令行工具设置/更新密码 | P0 |

### 2.2 详细描述

#### FR-01 全局请求拦截

**描述：** 以 Axum middleware 形式拦截所有进入的 HTTP 请求，根据认证状态决定放行或拒绝。

**验收标准：**
- 已认证用户（Cookie 有效）：放行所有请求
- 未认证用户访问页面：302 重定向到 `/auth-wall/login`
- 未认证用户访问 `/api/*`：返回 `401 Unauthorized`
- `/auth-wall/*` 路径和 `/favicon.ico`：始终放行（不需要认证）

#### FR-02 独立登录页面

**描述：** 提供一个完整的 HTML 登录页面，以 Rust 字符串常量内嵌，无需外部静态文件。

**验收标准：**
- 深色主题，响应式布局，适配移动端
- 密码输入框支持显示/隐藏切换
- 登录失败时显示错误消息和剩余尝试次数
- 被锁定时显示等待时间提示
- 登录成功后自动跳转到首页 `/`

#### FR-03 安全密码存储

**描述：** 使用 Argon2id（当前 OWASP 推荐的密码哈希算法）对密码进行哈希处理。

**验收标准：**
- 密码哈希使用随机盐值（每次不同）
- 哈希结果以 PHC 标准格式存储
- 密码文件中不包含任何明文信息
- 提供 `hash_password()` 和 `verify_password()` 公共函数
- 单元测试覆盖正确密码验证、错误密码拒绝、无效哈希处理

#### FR-04 暴力破解防护

**描述：** 按客户端 IP 地址追踪连续失败的登录尝试，超过阈值后临时锁定。

**验收标准：**
- 默认允许 5 次失败尝试，可通过 `AUTH_WALL_MAX_ATTEMPTS` 配置
- 超过阈值后锁定该 IP，默认 5 分钟，可通过 `AUTH_WALL_LOCKOUT_SECS` 配置
- 锁定期间返回 `429 Too Many Requests`，附带剩余等待秒数
- 登录成功后立即重置该 IP 的失败计数
- 锁定自然过期后自动重置

#### FR-05 Cookie 会话管理

**描述：** 基于内存的会话令牌存储，通过 HTTP Cookie 传递。

**验收标准：**
- 会话令牌为 32 字节随机 hex 字符串（256 位熵）
- Cookie 属性：`HttpOnly`（防 XSS）、`SameSite=Strict`（防 CSRF）、`Path=/`
- 默认会话有效期 24 小时，可通过 `AUTH_WALL_SESSION_SECS` 配置
- 过期会话在新会话创建时自动清理
- 登出时删除服务端会话并清除客户端 Cookie

#### FR-07 优雅降级

**描述：** 当密码文件不存在或为空时，跳过所有认证检查。

**验收标准：**
- 密码文件不存在 → 所有请求直接放行
- 密码文件存在但内容为空或全为空白 → 所有请求直接放行
- 无需额外配置或手动禁用

#### FR-11 CLI 密码管理工具（`set-password`）

**描述：** 交互式命令行工具，用于设置或更新密码。

**验收标准：**
- 引导用户输入新密码并确认
- 密码长度不少于 6 个字符
- 两次输入不一致时提示重试
- 已有密码时提示确认是否更新
- 写入前验证哈希正确性（自检）
- 自动创建密码文件所在的目录
- 支持通过 `AUTH_WALL_PASSWORD_FILE` 环境变量指定路径

## 3. 非功能需求

### 3.1 安全性

| 项目 | 要求 |
|------|------|
| 密码哈希 | Argon2id，随机盐值，PHC 格式 |
| Cookie 防护 | HttpOnly（防 XSS）、SameSite=Strict（防 CSRF） |
| 暴力破解 | 基于 IP 的限速锁定 |
| 传输安全 | 支持 Secure Cookie 标记（配合 HTTPS） |
| 令牌强度 | 256 位随机令牌（密码学安全的随机数生成器） |

### 3.2 性能

| 项目 | 要求 |
|------|------|
| 中间件延迟 | < 1ms（仅内存 HashMap 查找 + Cookie 解析） |
| 内存占用 | 与活跃会话数成正比，每个会话约 ~100 字节 |
| 密码验证 | Argon2id 约 100-500ms（有意设计，防暴力破解） |

### 3.3 可用性

| 项目 | 要求 |
|------|------|
| 零配置启动 | 未设密码时自动放行，无需任何配置 |
| 掉电恢复 | 服务重启后会话失效，用户需重新登录（安全设计） |
| 错误提示 | 登录失败时显示清晰的中英文错误消息 |

### 3.4 兼容性

| 项目 | 要求 |
|------|------|
| Rust 版本 | nightly >= nightly-2025-12-04（Edition 2024） |
| Axum 版本 | 0.8+ |
| 操作系统 | Linux / macOS / Windows |
| 浏览器 | 所有现代浏览器（Chrome / Firefox / Safari / Edge） |

### 3.5 独立性

| 项目 | 要求 |
|------|------|
| 内部依赖 | 零（不依赖 Vibe Kanban 的任何 crate） |
| 外部依赖 | 仅标准 Rust 生态库（axum, tokio, serde, argon2 等） |
| 构建 | 可独立 `cargo build` / `cargo test` |
| 仓库 | 独立 Git 仓库 `yangfei957/vibe-kanban-plus` |

## 4. 接口规格

### 4.1 HTTP API

| 方法 | 路径 | 说明 | 请求体 | 响应 |
|------|------|------|--------|------|
| `GET` | `/auth-wall/login` | HTML 登录页面 | — | `200` HTML |
| `POST` | `/auth-wall/api/login` | 验证密码 | `{ "password": "..." }` | `200` / `401` / `429` / `503` |
| `POST` | `/auth-wall/api/logout` | 登出 | — | `200` + 清除 Cookie |
| `GET` | `/auth-wall/api/status` | 查询状态 | — | `200 { authenticated, password_configured }` |

### 4.2 Rust 公共 API

```rust
/// 配置结构体
pub struct AuthWallConfig {
    pub password_hash_path: PathBuf,
    pub max_failed_attempts: u32,
    pub lockout_duration_secs: u64,
    pub session_expiry_secs: u64,
    pub secure_cookie: bool,
}

/// 共享状态（Clone-able，用于 Axum State）
pub struct AuthWallState { ... }
impl AuthWallState {
    pub fn new(config: AuthWallConfig) -> Self;
    pub fn is_password_configured(&self) -> bool;
}

/// 注册 auth-wall 路由
pub fn auth_wall_routes(state: AuthWallState) -> Router;

/// 认证中间件
pub async fn auth_wall_middleware(
    State(state): State<AuthWallState>,
    req: Request<Body>,
    next: Next,
) -> Response;

/// 密码工具函数
pub fn hash_password(password: &str) -> String;
pub fn verify_password(password: &str, hash_str: &str) -> bool;
```

### 4.3 环境变量

| 变量名 | 类型 | 默认值 | 说明 |
|--------|------|--------|------|
| `AUTH_WALL_PASSWORD_FILE` | 文件路径 | `<data_dir>/auth_wall_password.hash` | 密码哈希文件路径 |
| `AUTH_WALL_MAX_ATTEMPTS` | 正整数 | `5` | 锁定前最大失败尝试次数 |
| `AUTH_WALL_LOCKOUT_SECS` | 正整数 | `300` | IP 锁定持续时间（秒） |
| `AUTH_WALL_SESSION_SECS` | 正整数 | `86400` | 会话有效期（秒） |
| `AUTH_WALL_SECURE_COOKIE` | `true`/`false` | `false` | 是否设置 Secure Cookie |

## 5. 约束与假设

### 5.1 约束

- **单密码模式**：当前仅支持一个全局密码，不支持多用户/多密码
- **内存会话**：会话存储在内存中，服务重启后所有会话失效
- **无注册功能**：密码只能通过 CLI 工具离线设置，不提供 Web 注册界面
- **IP 来源**：依赖 `X-Forwarded-For` 请求头获取客户端 IP，需确保反向代理正确设置

### 5.2 假设

- 用户有权访问服务器命令行以运行 `set-password` 工具
- 如果通过反向代理部署，代理会正确设置 `X-Forwarded-For` 头
- 密码哈希文件具有适当的文件系统权限（仅服务进程可读写）

## 6. 未来演进方向

以下功能不在 v0.1.0 范围内，但可作为后续版本的方向：

| 方向 | 说明 |
|------|------|
| 多用户支持 | 支持多个用户名/密码对 |
| 持久化会话 | 将会话存储到文件或数据库，重启后保留 |
| 2FA/TOTP | 支持基于时间的一次性密码 |
| OAuth 集成 | 支持第三方登录（GitHub、Google 等） |
| Web 密码管理 | 在 Web 界面中修改密码（需先验证旧密码） |
| 审计日志 | 记录所有登录/登出事件到持久化日志 |
